# Accessibility testing

This page documents the three layers of automated a11y checks in this repo. Each layer has a different responsibility and a different blast radius for failure.

| Layer | Owns | Trigger | Gate |
| --- | --- | --- | --- |
| `eslint-plugin-jsx-a11y` | Static JSX patterns (missing `alt`, bad `role`, click-without-keyboard, etc.) | `lint.yml` on PR | Blocks PR (existing lint gate) |
| Lighthouse CI | Full-page audit at initial DOM of unauth + auth URLs; score-based gates; sticky PR comment | `pull_request` (annotate), `workflow_dispatch` (block), nightly schedule | Annotate on PR; block when invoked manually |
| Playwright + axe-core | Interactive states: modals open, side panels open, toasts visible, validation errors, focus traps | Existing `e2e-tests` matrix on PR | Blocks PR on `serious`/`critical` violations |
| `vitest-axe` | Component-level a11y regressions before they ship to CI | Existing `unit-tests` job on PR | Blocks PR |

axe-core is the rule engine Lighthouse uses internally for its accessibility category — rule overlap is intentional. The difference is *where in the page lifecycle* each layer runs and *who owns the gate*.

## Lighthouse CI

### What we check

The Lighthouse **accessibility** category is run against a fixed list of URLs served by the Vite dev server. The audit matrix and the per-level floor are defined in [.github/lighthouse/audit-sets.cjs](../../.github/lighthouse/audit-sets.cjs); each entry carries an inline comment naming the WCAG success criterion it most closely covers. We run Level **AA** by default (floor `0.90`), with a Landscape-specific `heading-order` override pinned to `warn` while the Vanilla Framework heading-structure migration is in flight.

### How CI invokes it

[.github/workflows/lighthouse-a11y.yml](../../.github/workflows/lighthouse-a11y.yml) calls the reusable workflow three ways:

| Trigger | Mode | URL set | Comment | Notes |
| --- | --- | --- | --- | --- |
| `pull_request` to `main` | `annotate` | `urls.public.txt` | Yes — sticky | Never blocks merge during the soak period. |
| `workflow_dispatch` | `block` or `annotate` (caller choice) | `public` / `msw` / `both` (caller choice) | No | Use this to dry-run a stricter gate before flipping the PR job. |
| `schedule` (nightly 03:30 UTC) | `annotate` | `both` | No | Catches regressions on URLs we don't scan on every PR. |

### How to run locally

```bash
pnpm install
pnpm dev                                # terminal A — keeps running on :5173
LHCI_AUTH_MODE=public pnpm a11y:ci      # terminal B — exits 0 if at/above floor
```

No `pnpm build` step: we scan the Vite dev server directly so a single edit-save-rerun loop is enough to iterate on a regression. Override the level or score floor by setting `LHCI_WCAG_LEVEL` (`a` / `aa` / `aaa`) or `LHCI_MIN_SCORE` (`0`..`1`).

### Interpreting a failure

1. Open the PR's `Lighthouse a11y / pr-public` check and read the sticky comment — it lists the failing URL, the score, and the top failing audit IDs with their WCAG mapping.
2. Download the `lighthouse-report-pr` artifact attached to the workflow run and open the HTML report for the specific URL. Each failing audit links to the offending DOM node and a remediation page.
3. Reproduce locally with the commands above, then iterate.

### Tuning

Edits land in [lighthouserc.cjs](../../lighthouserc.cjs):

- **Override an audit** — `config.ci.assert.assertions["audit-id"] = ["warn"]` (or `["error", { minScore: 0.9 }]`). We already ship one such line for `heading-order`.
- **Change the floor** — pass `minScore` to `createBaseConfig` (or set `LHCI_MIN_SCORE` for an ad-hoc run).
- **Add a URL** — append to `publicUrls` (or `authenticatedUrls` for `LHCI_AUTH_MODE=msw`).

Long-lived overrides should carry a brief comment explaining the reason and a planned removal trigger.

## Component-level (`vitest-axe`)

### When to add one

Add a component-level a11y test whenever you ship — or substantially alter — a component that controls focus, hides content from assistive technology, or builds its own DOM scaffolding rather than composing existing primitives. The standing list:

- Modals and confirmation dialogs (`Modal`, `ConfirmationModal`, anything using `role="dialog"`).
- Side panels (`SidePanel`, `SidePanelFormButtons`).
- Navigation / sidebar shells.
- Any new component that renders an `aria-*` attribute or sets `tabindex` directly.

These tests live in [src/tests/a11y/](../../src/tests/a11y/) — one file per component, named `<Component>.a11y.test.tsx`.

### Usage pattern

```tsx
import { axe } from "vitest-axe";
import { renderWithProviders } from "@/tests/render";
import MyDialog from "@/components/.../MyDialog";

it("has no detectable axe violations", async () => {
  const { container } = renderWithProviders(<MyDialog open />);
  expect(await axe(container)).toHaveNoViolations();
});
```

The `toHaveNoViolations` matcher is registered in [src/tests/setup.ts](../../src/tests/setup.ts) so individual tests don't need to call `expect.extend` themselves.

### Scoping or silencing rules

Disable a single rule for a known false positive:

```tsx
expect(
  await axe(container, { rules: { "color-contrast": { enabled: false } } }),
).toHaveNoViolations();
```

Run against a subtree instead of the whole rendered DOM by passing a more specific element instead of `container`. Disable as narrowly as possible — never blanket-disable a rule globally; that hides regressions.

### Layer overlap

axe-core is the same engine Lighthouse uses internally for its accessibility category — rule overlap with the Lighthouse layer is intentional. The component layer catches regressions before they reach a built page; Lighthouse catches the integration-level shape (heading order, landmarks, document-wide contrast). Both layers staying green is the contract.
