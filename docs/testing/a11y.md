# Accessibility testing

This page documents the three layers of automated a11y checks in this repo. Each layer has a different responsibility and a different blast radius for failure.

| Layer | Owns | Trigger | Gate |
| --- | --- | --- | --- |
| `eslint-plugin-jsx-a11y` | Static JSX patterns (missing `alt`, bad `role`, click-without-keyboard, etc.) | `lint.yml` on PR | Blocks PR (existing lint gate) |
| Lighthouse CI | Full-page audit at initial DOM of unauth + auth URLs; score-based gates; sticky PR comment | `pull_request` (annotate), `workflow_dispatch` (block), nightly schedule | Annotate on PR; block when invoked manually |
| Playwright + axe-core | Interactive states: modals open, side panels open, toasts visible, validation errors, focus traps | Existing `e2e-tests` matrix on PR | Blocks PR on `serious`/`critical` violations |
| `vitest-axe` | Component-level a11y regressions before they ship to CI | Existing `unit-tests` job on PR | Blocks PR |

axe-core is the rule engine Lighthouse uses internally for its accessibility category — rule overlap is intentional. The difference is *where in the page lifecycle* each layer runs and *who owns the gate*.
